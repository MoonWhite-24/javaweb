## A — 后端架构与数据层

### 1. Spring Boot 多模块架构

**讲解要点：** 项目采用 Maven 多模块分层架构，职责清晰。

```
market-common → market-dal → market-service → market-api → market-boot
     ↑              ↑              ↑              ↑            ↑
  公共类型       数据访问        业务逻辑       REST控制器     启动入口
```

**可讲内容：**
- 为什么分层？解耦、可维护、可测试
- 模块依赖关系：common 被所有模块依赖，dal 依赖 common，service 依赖 dal...
- 每层职责：
  - `market-common`：`R<T>` 统一响应、`UserDTO`、异常类、枚举
  - `market-dal`：MyBatis Mapper 接口 + XML 映射文件
  - `market-service`：业务逻辑 `@Service`
  - `market-api`：REST 控制器 `@RestController`
  - `market-boot`：`Application.java` 启动类 + `application.yml` 配置

**代码示例（可展示）：**
```java
// R<T> 统一响应封装
public class R<T> {
    private int code;
    private String msg;
    private T data;
    public static <T> R<T> ok(T data) { return new R<>(200, "success", data); }
    public static <T> R<T> error(String msg) { return new R<>(500, msg, null); }
}
```

## B — 核心业务与中间件

### 1. JWT 认证与权限控制

**讲解要点：** 双 Token 机制 + Filter/Interceptor 双重验证。

**认证流程：**
```
用户登录 → 生成 AccessToken(15min) + RefreshToken(7天)
    ↓
请求携带 Authorization: Bearer <token>
    ↓
JwtAuthFilter 拦截 → 验证Token → 提取UserDTO → 存入request
    ↓
AdminInterceptor → 检查 role==1（管理员）
    ↓
Controller 处理请求
```

**代码展示：**
```java
// JwtAuthFilter.java - JWT 认证过滤器
public class JwtAuthFilter extends OncePerRequestFilter {
    @Override
    protected void doFilterInternal(HttpServletRequest request, ...) {
        String header = request.getHeader("Authorization");
        if (header == null || !header.startsWith("Bearer ")) {
            sendError(response, 401, "未登录");
            return;
        }
        String token = header.substring(7);
        Claims claims = jwtUtil.parseToken(token);
        UserDTO user = new UserDTO();
        user.setId(jwtUtil.getUserId(claims));
        user.setRole(jwtUtil.getRole(claims));
        request.setAttribute("currentUser", user);
        chain.doFilter(request, response);
    }
}
```

```java
// AdminInterceptor.java - 管理员权限拦截器
public class AdminInterceptor implements HandlerInterceptor {
    @Override
    public boolean preHandle(HttpServletRequest request, ...) {
        UserDTO user = (UserDTO) request.getAttribute("currentUser");
        if (user == null || user.getRole() != 1) {
            response.setStatus(403);
            response.getWriter().write("{\"code\":403,\"msg\":\"无管理员权限\"}");
            return false;
        }
        return true;
    }
}
```

**白名单路径（无需登录）：**
- `/api/auth/login`, `/api/auth/register`
- `/api/products/*`, `/api/categories/*`, `/api/seckill/products`

### 2. Redis/Valkey 缓存策略

|----------|------|-----|
| `seckill:stock:{id}` | 秒杀库存（原子操作） | 活动结束 |

**缓存更新策略：**

- 秒杀库存使用 Lua 脚本原子操作

### 3. Kafka 异步订单处理

**讲解要点：** 秒杀成功后通过 Kafka 异步创建订单，削峰填谷。

**流程：**
```
秒杀成功 → 生成订单号 → 发送 Kafka 消息 → OrderConsumer 消费 → 写入数据库
```

**代码展示：**
```java
// 生产者：秒杀成功后发送消息
kafkaTemplate.send(KafkaTopic.ORDER_CREATE, JsonUtil.toJson(orderMsg));

// 消费者：异步处理订单创建
@KafkaListener(topics = KafkaTopic.ORDER_CREATE, groupId = "market-order-processor")
public void onOrderCreate(String message) {
    Map<String, Object> msg = JsonUtil.fromJson(message, Map.class);
    Long orderNo = Long.valueOf(msg.get("orderNo").toString());
    processSeckillOrder(orderNo, userId, seckillProductId);
}
```

**幂等性保证：** 使用 Redis `setIfAbsent` 防止重复消费
```java
String idempotentKey = RedisKeyPrefix.ORDER_CREATED + orderNo;
Boolean acquired = redisTemplate.opsForValue().setIfAbsent(idempotentKey, "1", 1, TimeUnit.HOURS);
if (Boolean.FALSE.equals(acquired)) return; // 重复消息，忽略
```

### 4. 秒杀系统（核心亮点）

**讲解要点：** 高并发场景下的库存扣减，Lua 脚本保证原子性。

**秒杀流程：**
```
用户请求 → 检查活动时间 → Redis Lua 脚本原子操作 → 发送 Kafka 消息 → 返回结果
```

**Lua 脚本（`seckill_stock.lua`）：**
```lua
local stock = redis.call('GET', stock_key)
if not stock or tonumber(stock) <= 0 then
    return -1  -- 库存不足
end
if redis.call('SISMEMBER', users_key, user_id) == 1 then
    return -2  -- 重复抢购
end
redis.call('DECR', stock_key)          -- 原子扣减库存
redis.call('SADD', users_key, user_id) -- 记录用户
return 1  -- 成功
```

**为什么用 Lua 脚本？**
- Redis 单线程执行，保证原子性
- 避免超卖问题
- 比数据库乐观锁性能高 10 倍以上

### 2. Axios 请求拦截器

**讲解要点：** 自动注入 JWT Token，401 自动刷新。

```javascript
// request.js - Axios 拦截器
// 请求拦截：自动添加 Token
request.interceptors.request.use(config => {
    const token = localStorage.getItem('accessToken')
    if (token) config.headers.Authorization = `Bearer ${token}`
    return config
})

// 响应拦截：401 自动刷新 Token
request.interceptors.response.use(
    response => response,
    async error => {
        if (error.response?.status === 401 && !error.config._retry) {
            error.config._retry = true
            const refreshToken = localStorage.getItem('refreshToken')
            const { data } = await axios.post('/api/auth/refresh', { refreshToken })
            localStorage.setItem('accessToken', data.data.accessToken)
            error.config.headers.Authorization = `Bearer ${data.data.accessToken}`
            return request(error.config)
        }
        return Promise.reject(error)
    }
)
```

### 3. Vue Router 路由守卫

**讲解要点：** 前端路由级别的权限控制。

```javascript
router.beforeEach((to, from, next) => {
    const token = localStorage.getItem('accessToken')
    const userInfo = JSON.parse(localStorage.getItem('userInfo') || 'null')

    // 需要登录但没有 Token → 跳转登录页
    if (to.meta.requiresAuth && !token) return next('/login')

    // 需要管理员权限但不是管理员 → 跳转首页

    next()
})
```

## 常见答辩问题及参考回答

### 架构类

**Q: 为什么用多模块架构？**
> A: 分层解耦，common 提供公共类型，dal 负责数据访问，service 处理业务逻辑，api 定义接口，boot 启动配置。每层只依赖下层，便于维护和测试。

**Q: 前后端分离的好处？**
> A: 前端独立开发部署，后端专注 API，通过 RESTful 接口通信。支持多端复用（Web、小程序），开发效率高。

### 数据库类

**Q: 为什么用 MyBatis 而不是 JPA？**
> A: MyBatis 支持原生 SQL，灵活性高，便于优化复杂查询。XML 方式管理 SQL 语句，便于维护。项目有大量动态查询（商品搜索、订单筛选），MyBatis 的 `<if>` 动态 SQL 更适合。

**Q: 如何防止 SQL 注入？**
> A: MyBatis 使用 `#{}` 参数占位符，自动预编译，防止 SQL 注入。所有用户输入都通过参数传递，不拼接 SQL。

### 安全类

**Q: JWT Token 过期怎么办？**
> A: 双 Token 机制。AccessToken 15 分钟过期，RefreshToken 7 天有效。前端 axios 拦截器检测到 401 时自动用 RefreshToken 换取新 AccessToken，用户无感知。

**Q: 如何防止越权访问？**
> A: 两层防护。JwtAuthFilter 验证 Token 合法性并提取用户信息，AdminInterceptor 检查 role==1 管理员权限。前端路由守卫也做了同样的检查，双重保障。

### 高并发类

**Q: 秒杀怎么防止超卖？**
> A: 使用 Redis Lua 脚本原子操作。脚本内先检查库存，再扣减，再记录用户，整个过程在 Redis 单线程中原子执行，不会出现并发问题。

**Q: 为什么用 Kafka 异步处理订单？**
> A: 秒杀瞬间高并发，如果同步写数据库会成为瓶颈。Kafka 削峰填谷，秒杀请求先入队，后台异步消费处理，保护数据库。同时支持重试和幂等性。

### 部署类

**Q: 3 个节点怎么保证数据一致？**
> A: 数据库和 Redis 是共享的（master 节点），3 个应用节点连接同一个 MariaDB 和 Valkey。Snowflake 每个节点配置不同 worker-id，保证 ID 唯一。

**Q: Nginx 负载均衡策略？**
> A: 默认轮询（round-robin），3 个节点平均分配请求。支持权重配置，可根据节点性能调整。
